对话玩具故事机

发表于 2025-09-15 更新于 2025-09-17 分类于 code 阅读次数： 本文字数： 3.4k 阅读时长 ≈ 12 分钟

前言

目前玩具的讲故事机制处于萌芽阶段，现在的方案是让LLM即兴创作，会有一个比较严重问题：生成的故事的AI味很重，无论什么故事感觉都一个套路，现在就这个问题进行优化，目前的思路是将故事存到数据库中（本文用sqlite），然后通过text2sql技术实现自然语言到数据库查询的链接。更具体的流程是：小朋友让讲个故事（或者讲个关于什么什么的故事），大模型识别到是让讲故事，就调用讲故事的工具，这个讲故事的工具是一个基于LLM的text2sql工具，将小朋友的自然语言指令翻译成sql指令进行数据库的查询，然后查询到的故事返回给最初的LLM，实现LLM基于素材库讲故事而不是完全即兴讲故事，当然如果数据库中没有小朋友想听的故事，那么依旧即兴创作。

本文将从数据库基础讲起，逐步实现上述功能。

正文

数据库基础

  • 数据库（Database）：一个专门用来放很多结构化数据的大柜子（stories.db 就是一个柜子文件）。

  • 表（Table）：柜子里的抽屉（比如 stories 抽屉），抽屉里装的是同类的记录。

  • 行 / 记录（Row / Record）：抽屉里的每一份文件（一条故事）。

  • 列 / 字段（Column / Field）：每份文件的标签（比如 title、content），每个标签有固定的类型或规则。

  • 主键（PRIMARY KEY）：用来唯一标识每份文件的编号（类似身份证号）。

SQLite 的存储类型只有几种“存储类”：

  • NULL、INTEGER（整数）、REAL（浮点数）、TEXT（文本）、BLOB（二进制）
    注意：SQLite 的类型是“灵活类型（type affinity）”，你声明 TEXT/INTEGER 是对数据做建议，SQLite 仍能存别的东西（但按规则尝试转换）。

逐项解释：

  1. CREATE TABLE
    • 含义：创建一张表（抽屉）。
    • 等价中文：创建表格。
  2. IF NOT EXISTS
    • 含义：只有当该表还不存在时才创建，如果已存在就不报错。
    • 作用：避免重复建表时报错（方便脚本多次运行）。
  3. stories
    • 含义：表名（你抽屉的名字）。由字母/数字/下划线组成，大小写通常不敏感。
  4. ( 与 ) 与逗号 ,
    • 括号内列出了每一列（字段）的定义。每列定义之间用逗号分隔。

接下来按列逐个详细拆解：

id INTEGER PRIMARY KEY AUTOINCREMENT,

  • id
    • 列名（字段名）。你可以把它当作“label = id”。
  • INTEGER
    • 声明的数据类型（整数）。在 SQLite 中表示整数字段或带整形亲和力。
  • PRIMARY KEY
    • 约束（constraint）。表示这个列是表的主键，每一行必须唯一地由这个字段识别。
    • 在 SQLite 中，如果主键是 INTEGER PRIMARY KEY，它会变成表的 rowid 的别名（也就是底层的行号/唯一 id，SQLite 内部特殊处理，性能最好）。
  • AUTOINCREMENT
    • 修饰词（keyword）。告诉 SQLite 每插入一行时，自动分配一个比之前最大的 id 更大的整数。
    • 注意区别：
      • SQLite 如果你写 INTEGER PRIMARY KEY（不加 AUTOINCREMENT），它也会自动给你一个自增的 rowid（通常是上一最大值 + 1）。
      • AUTOINCREMENT 会额外保证不会重用以前使用过的 rowid（即使表里有删除过的行），并且让分配策略更严格（记录在 sqlite_sequence 表）。但它也带来一点性能/空间代价，所以多数情况下 INTEGER PRIMARY KEY 就够了，只有需要“绝对不重用历史 id 且保证严格单调递增”时才用 AUTOINCREMENT。

总结一句话：
id INTEGER PRIMARY KEY AUTOINCREMENT 的含义是：

列名叫 id，它是整数类型、表的唯一标识（主键），并且让数据库自动分配递增的数值作为 id。

title TEXT NOT NULL,

  • title：字段名（故事标题）
  • TEXT：文本类型
  • NOT NULL：约束，表示该字段不能为空（插入时必须提供值）

theme TEXT,

  • theme：字段名（主题）
  • TEXT：可以存文本
  • （没有 NOT NULL，因此允许为空）

keywords TEXT,

  • keywords：字段名。关键字用逗号分隔存在这列（例：”兔子，乌龟，赛跑”）。
  • 注意：这是一种简单做法，方便但不利于做复杂检索（更复杂的做法是建立关联表）。

content TEXT NOT NULL,

  • content：故事正文
  • TEXT：文本
  • NOT NULL：必填

origin TEXT,

  • origin：来源（例如“伊索寓言“，“原创”）

language TEXT DEFAULT 'zh',

  • language：语言字段
  • TEXT：文本
  • DEFAULT 'en'：如果插入时没有提供 language 值，默认就插入 'zh'（即中文）。DEFAULT 可以是字符串、数字或像 CURRENT_TIMESTAMP 这样的关键字（取决字段）

created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

  • created_at：创建时间
  • TIMESTAMP：在 SQLite 中这其实是一种“语义上的类型名”（底层仍以 TEXT/NUMERIC 存），但它表达你的意图：这是时间戳字段。
  • DEFAULT CURRENT_TIMESTAMP：如果插入时不提供值，SQLite 会把当前时间（UTC）写入（格式通常是 YYYY-MM-DD HH:MM:SS）。
    • 注意：DEFAULT 只在 插入（INSERT）时生效，之后该列不会自动变化（要自动更新时间需在 UPDATE 时显式设置或由应用/触发器处理）。

updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP

  • 同 created_at，但语义是记录最后更新时间。通常你需要在每次修改（UPDATE）记录时把 updated_at 更新为 CURRENT_TIMESTAMP，这可以由程序代码在执行 UPDATE 时带上，或用 DB 的触发器（trigger）实现自动更新。

结合简单的Python 代码进行讲解

逐行说明（重要行）：

  • import sqlite3
    • 加载 Python 自带的 SQLite3 模块，用来操作 SQLite 数据库文件。
  • conn = sqlite3.connect("stories.db")
    • 连接或创建数据库文件 stories.db（相对路径，文件在当前工作目录）。
    • 返回 Connection 对象 conn。
    • 你也可以用 ":memory:" 创建临时内存数据库（程序结束就没了）。
  • cursor = conn.cursor()
    • 创建一个 Cursor 对象，用它来执行 SQL 语句并取回结果。
  • cursor.execute("""CREATE TABLE IF NOT EXISTS stories (...)""")
    • 执行建表语句（把前面逐词解释的语句传给 DB 引擎）。
    • 多行字符串用 """..."""。
  • cursor.execute("""INSERT INTO stories (...) VALUES (?, ?, ?, ?, ?, ?)""", (...))
    • ? 是参数占位符（参数化查询）。
    • 第二个参数是一个元组，按顺序对应每个 ?。
    • 为什么用 ? 而不是把字符串拼起来？ —— 用 ? 可以防止 SQL 注入攻击，并且让 sqlite3 模块做好适当的转义与类型转换。
    • 例如 ("龟兔赛跑", "坚持", ...) 会被安全地插入。
  • conn.commit()
    • 把当前连接的事务提交（写入磁盘）。
    • 如果你不 commit()，在多数情况下当连接关闭时会回滚（不保存），所以写操作后必须 commit()。
  • cursor.execute("SELECT title, content FROM stories WHERE keywords LIKE ?", ('%' + keyword + '%',))
    • SELECT：查询语句。
    • keywords LIKE ?：LIKE 是模糊匹配运算符，% 在 SQL 中是任意数量字符的通配符。'%乌龟%' 表示关键词中包含“乌龟”的行。
    • 第二个参数 ('%' + keyword + '%',) 是传给 ? 的实际值。
  • results = cursor.fetchall()
    • fetchall() 返回所有查询结果，类型是 list，每个元素是 tuple（对应 SELECT 列顺序）。
    • 例如 results = [('龟兔赛跑', '从前有...'), (...)]
  • conn.close()
    • 关闭数据库连接（释放资源）。在脚本结束前最好显式关闭。

额外重要知识点（实际使用时很关键）

  1. 事务（Transaction）

    • 写操作（INSERT/UPDATE/DELETE）通常在事务中执行。commit() 提交事务；rollback() 回退。
    • Python 的 sqlite3 默认会自动在你执行写操作时隐含 START TRANSACTION（事务），直到你 commit()。

  2. 参数化查询 & 安全

    • 永远使用参数 (?)，不要把用户输入直接拼进 SQL 字符串（会导致注入风险）。

  3. 索引（Index）

    • CREATE INDEX idx_theme ON stories(theme);
    • 索引能加速按该列查询（WHERE theme=…），但会稍微增加写入时的开销和占用空间。
    • 建议对常用过滤字段建立索引（但别给每列都建索引）。

  4. LIKE 的大小写

    • 在 SQLite 中 LIKE 对 ASCII 字符通常不区分大小写；可以用 COLLATE NOCASE 做更多控制。

  5. DEFAULT CURRENT_TIMESTAMP 的时区

    • SQLite 的 CURRENT_TIMESTAMP 返回 UTC 的时间字符串 YYYY-MM-DD HH:MM:SS。
    • 如果你需要本地时区（如 +08:00），在应用层处理时间字符串或在插入时用 Python 的本地时间。

  6. updated_at 的自动更新

    • DEFAULT 只在插入时填值；updated_at 不会自动在 UPDATE 时变化。

    • 做法：

      • 在应用层每次 UPDATE 时带上 updated_at=CURRENT_TIMESTAMP；
      • 或者在数据库写 TRIGGER（触发器）自动赋值（触发器语法更复杂，需要小心）。

    • 命令行：sqlite3 stories.db 然后 .tables 查看表，执行 SQL。

  8. 性能与拆表建议

    • 现在把 keywords 存为逗号分隔的字符串（简单）。如果将来要做复杂检索或标签统计，建议做一对多的关联表（story_keywords）或使用向量检索 / RAG 等方式。

六、常见错误与如何避免

  • 忘记 conn.commit() → 写入没有保存。
  • 把用户输入直接拼接进 SQL → SQL 注入漏洞。
  • 以为 DEFAULT 会在每次 UPDATE 自动刷新 → 不会，需显式 UPDATE。
  • 误用 AUTOINCREMENT：通常不必要且会增加开销。
