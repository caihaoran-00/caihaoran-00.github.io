聊天历史管理&MemU原理解析

发表于 2025-09-24 更新于 2025-09-30 分类于 code 阅读次数： 本文字数： 22k 阅读时长 ≈ 1:22

前言

随着玩具的不断优化迭代，认为对聊天历史的管理需要下些功夫。目前并没有对长期记忆做文章，只有短期记忆（使用最近的五轮对话），导致的最大的问题是每次新对话都是没有任何记忆的，是“从0开始”的，本文将记录对记忆系统的思考以及对MemU的原理解析，目标包括两方面，一是怎么科学经济的管理聊天历史，二是MemU的原理有可借鉴的地方吗。

正文

在正式开始之前，先明确下本文所说的短期记忆和长期记忆所指：

短期记忆 = 当前对话轮次的上下文（几轮对话内容）
长期记忆 = 用户与AI历史交互中沉淀的个性化信息（偏好、习惯、重要事件、性格画像等）

一般各自的实现方式如下：
短期记忆：

  • 直接拼接上下文：把最近 N 轮对话（用户+AI）作为 prompt 的一部分传给模型。
  • 滑动窗口机制：只保留最近 K 个 token 或最近 M 条消息。
  • 摘要压缩（进阶）：对前面多轮对话做摘要，节省 token，保留语义。

理论上摘要压缩方案的上限最高，但实际操作下来发现这种方式不确定性较大，且成本较高，最终出于效果与成本均衡考虑，选择最近的5段对话作为短期记忆。

长期记忆：

  • 向量数据库（如 Pinecone, Milvus, Weaviate）：存储用户历史对话片段/关键事件的 embedding，按需检索。
  • 图数据库（如 Neo4j）：存储“用户-AI”之间的关系网络（如“喜欢科幻电影”、“讨厌香菜”、“妈妈叫李女士”）。
  • 结构化知识库：用 JSON / SQL 存储用户档案（User Profile）。
  • 记忆摘要 + 时间戳：定期对历史对话生成摘要，并打标签归档。

这就是本文重点探索的内容，目前较为出名的长期记忆管理工具包括但不限于：

  • zep：2023-4-17发布，3.7k Star

  • mem0：2023-06-12发布，40.4k Star

  • Letta（原 MemGPT）：2023-10-08发布，18.5k Star

  • memori：2025-7-20，1.1k Star

  • memU：2025-8-3发布，2.3k Star

单从上面数据来看，大家对mem0、 Letta（原 MemGPT）、memU的认可度高一些，且memU作为目前的较新开源项目，介绍中称：MemU 是一个为 AI 伴侣设计的开源记忆框架——高精度、快速检索、低成本。它充当一个智能的“记忆文件夹”，适应不同的 AI 伴侣场景。使用 memU，你可以构建真正记得你的 AI 伴侣。它们学习你是谁，你关心什么，并通过每一次互动与你共同成长。

我决定先从memU入手，学习其原理，以求将其关键思路整合到我们的玩具的记忆系统中，Let’s go。

🎯 业务流程总结：

这个脚本完整模拟了一个“对话 → 记忆 → 检索”的闭环：

  1. 先确认服务活着（health check）
  2. 把一段对话“喂”给系统记忆（memorize）
  3. 等待系统异步处理完成（polling）
  4. 查看系统自动生成了哪些记忆分类（default categories）
  5. 用自然语言搜索相关记忆内容（related memories）
  6. 用自然语言搜索相关记忆分类（clustered categories）

→ 适合测试“长期记忆系统”的核心功能是否正常。

作用：

参数说明：

参数 说明

生产建议：不要用 *，应明确指定前端域名，如 ["https://your-app.com"]

💡 中间件执行顺序：CORS → Logging → 路由处理 → Logging → CORS（请求进、响应出）

函数定义（装饰器 + 签名）

  1. @router.post("/memorize", response_model=MemorizeResponse)

  • @router.post：这是一个 FastAPI 路由装饰器，表示：

    “当有人发送 POST 请求 到 /memorize 路径时，就运行我下面这个函数！”

  • response_model=MemorizeResponse：

    “无论函数返回什么，最终返回给用户的 JSON 结构，必须符合 MemorizeResponse 这个模型。”
    （比如必须包含 task_id, status, message 字段）

  1. async def memorize_conversation(...)
  • async def：这是一个 异步函数，意味着它可以在等待 I/O（如网络请求、文件读写）时“让出控制权”，让服务器同时处理其他请求。
  • 函数名 memorize_conversation：清晰表达功能 —— “记忆对话”
  1. 参数列表
参数 类型 作用 如何传入？
request: MemorizeRequest Pydantic 模型 用户发来的数据（对话内容、用户ID等） 自动从 HTTP 请求体（JSON）解析
background_tasks: BackgroundTasks FastAPI 工具类 用来启动后台任务 FastAPI 自动注入
memory_svc: MemoryService = Depends(get_memory_service) 服务类实例 记忆业务逻辑（提取、存储等） 通过 Depends(get_memory_service) 注入
task_svc: TaskService = Depends(get_task_service) 服务类实例 任务管理（创建、查询状态） 通过 Depends(get_task_service) 注入

✅ 关键概念：依赖注入（Dependency Injection）

  • 在这里最大的作用是实现单例，而不同每次调用该接口是都初始化创建MemoryService() 实例
  • 你不用自己 创建 MemoryService() 实例
  • FastAPI 会自动调用 get_memory_service()，把返回的实例塞进 memory_svc
  • 这样函数只关心“用服务”，不关心“怎么创建服务”

启动后台任务

这行代码到底做了什么？

它告诉 FastAPI：“等 HTTP 响应发出去后，请在后台悄悄运行 _process_memorization_task 函数，并传这些参数！”

为什么需要“后台任务”？

  • LLM 调用很慢（可能 2~5 秒）
  • 如果同步执行，用户会卡在“加载中…”很久
  • 异步处理：先返回 {"task_id": "xxx", "status": "PENDING"}，让用户去做别的事

后台任务函数 _process_memorization_task 会做什么？

  1. 调用 memory_svc.memorize_conversation(...) → 让 LLM 分析对话
  2. 提取记忆点（如“用户喜欢 Python”）
  3. 存储到记忆库
  4. 更新任务状态为 SUCCESS 或 FAILURE

这就是“异步写”的精髓：写操作慢 → 放后台；读操作快 → 直接返回。

_process_memorization_task

这个函数 _process_memorization_task 是 MemU 记忆系统的核心后台处理器，它负责真正执行耗时的记忆提取工作。

它是一个“后台工人”，专门负责：
1️⃣ 把用户提交的对话交给 LLM 分析
2️⃣ 提取出结构化记忆（如“用户喜欢 Python”）
3️⃣ 存储到记忆库
4️⃣ 更新任务状态（成功/失败）
而且全程在后台默默干活，不阻塞用户的 HTTP 请求。

def _load_actions(self):
    """Load all available actions from the registry"""
    for action_name, action_class in ACTION_REGISTRY.items():
        try:
            # 创建动作实例，传入memory_core
            action_instance = action_class(self.memory_core)

            # 将动作实例存储到self.actions字典中
            self.actions[action_name] = action_instance
          	logger.debug(f"Loaded action: {action_name}")
       	except Exception as e:
           	logger.error(f"Failed to load action {action_name}: {e}")

这里的ACTION_REGISTRY来源于from .actions import ACTION_REGISTRY

UpdateMemoryWithSuggestionsAction

核心功能：根据建议内容更新指定类别的记忆文件，支持四种操作类型。

四种操作类型的详细说明：

1. ADD（添加）
  • 功能：添加全新的记忆项到类别中
  • 使用场景：当建议包含现有类别中不存在的新信息时
2. UPDATE（更新）
  • 功能：修改现有记忆项的内容
  • 使用场景：当建议提供现有记忆项的更新或更详细信息时
3. DELETE（删除）
  • 功能：从类别中移除过时或不正确的记忆项
  • 使用场景：当发现现有记忆项已过时或错误时
4. TOUCH（触碰）
  • 功能：更新现有记忆项的时间戳，标记为最近访问过
  • 使用场景：当现有记忆项已经充分涵盖建议内容时，只需标记为活跃状态
  • 具体实现：在代码中看到注释”””should update “updated_at” “””，表示应该更新最后访问时间戳

总结

一句话概括：UpdateMemoryWithSuggestionsAction负责将LLM生成的建议内容实际应用到记忆文件中，通
过添加、更新、删除或触碰操作来维护记忆的准确性和时效性。

想象你有一本个人记忆笔记本，里面分了好几个章节，比如：

  • 【活动】：记录你每天干了啥（“今天和小明聊了Python”）
  • 【个人资料】：记录你的基本信息（“我叫张三，住在北京”）
  • 【事件】：记录重要日子（“下周二要开项目会”）

现在，你想给这些记忆加上“超链接”，让相关的内容能互相跳转。比如，看到“和小明聊Python”，能自动跳到“我叫张三”和“下周二要开项目会”，因为它们都相关。

link_all_items 就是决定你怎么加这些链接的两种模式：

路线一：link_all_items = False（默认模式）—— “单点突破”

  • 怎么干？
    你只给某一条特定的记忆加链接。
    比如，你只关心 “今天和小明聊了Python” 这一条，就只给它找关联。

  • 过程：

    1. 你告诉系统：“请给 memory_id=mem_123 这条记忆加链接”。
    2. 系统就只分析 这一条，然后去全本笔记本里（所有章节）找和它相关的内容。
    3. 找到后，只把链接加到 mem_123 这一行的末尾。

  • 适用场景：

    • 增量更新：你刚新增了一条记忆，想立刻给它建立关联。
    • 精准操作：你只想处理某一条，不想动其他的。

总结：一次只处理一条，精准、高效、省资源。

路线二：link_all_items = True —— “全面体检”

  • 怎么干？
    你让系统把整个章节里所有的记忆都检查一遍，给每一条都加上链接。

  • 过程：

    1. 你告诉系统：“请给 【活动】 这个章节里的所有记忆都加链接”。
    2. 系统就逐条遍历【活动】章节里的每一条记忆（可能是100条）。
    3. 对每一条，都重复“单点突破”的过程：分析它，去全本笔记本里找关联，然后把链接加到它自己那行末尾。

  • 适用场景：

    • 初次构建：你刚导入了一大堆旧记忆，需要一次性建立完整的知识网络。
    • 全局更新：你修改了向量模型或相似度算法，需要重新计算所有记忆的关联关系。
    • 定期维护：每周日半夜自动运行一次，确保所有记忆的链接都是最新的。

总结：一次性处理整个章节，全面、彻底，但非常耗时耗力。

两条路的联系

  1. 底层逻辑相同：
    无论是处理一条还是处理一百条，每一条记忆的关联查找过程（计算向量、跨分类搜索、筛选结果）是完全一样的。link_all_items=True 本质上就是把 link_all_items=False 的流程循环执行 N 次。

  2. 目标一致：
    两条路的最终目的都是丰富记忆之间的关联，让整个记忆系统从“一堆孤立的笔记”变成一个“互联互通的知识图谱”。

  3. 互补使用：

    • 平时用 “单点突破” (False) 来处理新记忆，保证实时性。
    • 定期用 “全面体检” (True) 来做全局优化，保证完整性。

一句话终极总结

link_all_items=False 是“外科手术刀”，精准处理单条记忆；
link_all_items=True 是“全身CT扫描”，彻底检查整个分类。
一个用于日常维护，一个用于大修大补。

总结

MemU是一个富有远见的AI伴侣记忆框架。它不满足于让AI“记住”，而是致力于让AI像人一样“理解和组织”自己的记忆。通过一系列由大语言模型（LLM）驱动的智能工具，它构建了一个能够自我演化、关联和抽象的动态记忆网络。

核心优势与能力：

  • 动态的自组织能力: 它最大的亮点是能够自动从零散的对话中“发现”并“创造”出新的主题（如programming_assistance.md），并将相关的记忆进行聚类。同时，它还能通过LinkRelatedMemoriesAction在不同记忆之间建立链接，将孤立的信息点编织成一张相互关联的知识网络，让AI具备联想和举一反三的能力。

  • 拟人化的记忆更新与遗忘: 它通过“读取-重写”的模式，让LLM在充分理解上下文后，智能地更新、修正甚至“遗忘”（在重写时省略）过时的信息。这模拟了人类记忆动态、非机械的特点，使AI伴侣的成长更为自然。

  • 灵活的分层检索: 框架提供了从“快速事实查询”到“深度语义搜索”再到“主题探索”的多种检索模式，使AI可以根据对话需要，灵活地调用不同层次的记忆，进行更有策略性的回应。

主要局限与挑战：

  • 高昂的运行成本: 其核心的“智能化”严重依赖大模型的实时分析和重写能力，这导致了极高的Token消耗。每一次记忆的整合都可能是一次昂贵的API调用，这是其大规模应用面临的最主要障碍。
  • 传统的可扩展性瓶颈: 当前基于文件系统和暴力向量搜索的实现，在面对长期交互所产生的海量数据时，会迅速遭遇性能瓶颈。若不引入专业的向量数据库和更高效的存储方案，系统将难以为继。
  • “黑盒”依赖与控制力缺失: 将记忆的组织、更新和遗忘等核心逻辑完全委托给LLM，使其行为具有不可预测性，调试困难。同时，框架缺少让用户或开发者能“手术刀式”精确删除或修改单条错误记忆的直接API，这在需要高度用户信任的伴侣场景中是一个显著的缺陷。

思考

目前记忆功能的实现思路是通过每日总结注入到提示词，但可以预见的是：

  • 总结的内容不够细，会遗漏内容（也确实不想让总结的太细）
  • 注入模式不灵活，直接一把注入，没有embedding后进行相似度计算机制
