二阶段AI课程：第一次作业

发表于 2025-12-19 更新于 2025-12-22 分类于 code 阅读次数： 本文字数： 4.1k 阅读时长 ≈ 15 分钟

前言

第二阶段的AI课程开始了，前一阶段主要是深度学习基础内容，从基础理论（导数、偏导、向量等）到反向传播再到手写识别。现在第二阶段开始了，内容为LLM相关，第一次课主要讲了部分大模型的基础，包括分词器、Embedding层、LLM输出的原理（预测下一个token）以及提示词相关，与一阶段一样，也是留了几道作业，下面我们一起来看下吧。

第二题

设计一个评估方案，评估您的玩具对话提示词效果（不同的维度不同的评估，请自由发挥）

答：

三层自动化评估方案：从代码检查到用户行为分析

目标：用自动化替代高成本、低效率的人工逐条评估，实现快速、经济、数据驱动的提示词优化。

第一层：规则引擎（自动筛选“硬伤”）

  • 做什么：用预设的规则，自动、快速地标记出有明显错误的回答。
  • 如何做：
    • 关键词黑名单：检查回答中是否包含“AI模型”、“语言模型”等暴露身份的词，或暴力、色情等危险词。
    • 正则表达式：检查回答格式是否正确，例如是否包含了要求的 [EMOTION:xxx] 标记。
    • 轻量级NLP模型：快速判断回答的情感是积极还是消极，或回答与问题的相关性。
  • 产出：一个简单的 “通过/失败” 标签。例如，is_safe: false 或 has_persona_leak: true。
  • 好比：工厂流水线上的初筛质检员，把有硬伤的次品直接挑出来。

第二层：AI裁判（智能评分“软实力”）

  • 做什么：用一个更强大的LLM（如GPT-4o）扮演“专业质检官”，对回答的“软质量”进行深度打分。
  • 如何做：
    1. 设计“裁判提示词”：写一个详细的指令，告诉“裁判LLM”它需要从哪些维度（如人设一致性、共情能力、年龄适配性、安全性等）进行1-10分的评分，并给出评语。
    2. 批量调用API：将你的测试用例（用户问题 + 玩具回答）批量发送给这个“裁判LLM”。
  • 产出：一个结构化的 JSON评分报告。
  • 好比：经验丰富的品控专家，对产品的细节和体验进行精细化评级。

第三层：用户行为分析（真实数据验证“最终效果”）

  • 做什么：分析真实用户的对话日志，通过统计关键行为指标，来评估提示词在真实世界中的表现。

  • 如何做：

    1. 定义关键指标：

      • 平均对话轮次 (ACR)：用户平均和玩具聊几轮？轮次越多，说明粘性越强。

      • 会话中断率：有多少用户在聊了1-2轮后就走了？中断率高说明初次体验差。

      • “重述”事件率：用户重复提问的频率有多高？高频率说明上一个回答质量差。

        通过向量相似度实现。

      • 安全护栏触发率：AI触发安全规则的频率是多少？

        通过构建提示词，若触发安全护栏使其输出固定标志。

    2. 技术实现：使用SQL和Python（计算语义相似度）从数据库中自动计算这些指标，并呈现在数据看板上。

  • 产出：可量化的 业务指标数据，如“V2.1版本的ACR为6.5，高于V2.0的5.8”。

  • 好比：市场调研报告，用真实数据告诉你产品在市场上是否受欢迎。

如何整合：一个高效的迭代闭环

这三层不是孤立的，而是形成了一个自动化的工作流：

  1. 开发阶段：
    • 写好新提示词后，用**第一层（规则引擎）**快速跑一遍，过滤掉硬伤。
    • 再用**第二层（AI裁判）**对剩余的样本进行深度评分，确保软质量达标。
  2. 灰度发布阶段：
    • 将通过测试的新版本，推送给1%的真实用户。
  3. 监控与决策阶段：
    • 密切关注**第三层（用户行为分析）**的数据看板，对比新旧版本的关键指标。
    • 如果数据变好（如ACR上升，中断率下降），则全量发布新版本。
    • 如果数据变差，则立即回滚，并回到第一步，结合第一层和第二层的输出来分析问题、进行优化。
      通过这个三层方案，你将评估工作从“阅读成千上万条对话”的苦力活，变成了“监控数据和评分报告”的智能化决策过程。

第三题

了解任一模型任一项评分背后的比较机制

答：
这些Benchmark（基准测试）是评估大语言模型（LLM）在不同能力维度上表现的标准测试集，它们就像”考试科目”，用来衡量模型在特定任务上的能力水平。详细科普：

  1. English（英语能力）
  • MMLU-Pro：大规模多任务语言理解测试，评估模型的知识广度和推理能力
  • GPQA Diamond：专业级科学问答，测试模型在复杂科学问题上的准确性
  • HLE：可能指某种语言理解或逻辑推理测试
  1. Code（编程能力）
  • LiveCodeBench：代码生成和调试能力测试
  • Codeforces：编程竞赛评分，评估算法和编程水平
  1. Math（数学能力）
  • AIME 2025：美国数学邀请赛，高级数学问题
  • HMMT：哈佛-麻省理工数学竞赛
  • IMOAnswerBench：国际数学奥林匹克相关测试
  1. Code Agent（代码智能体）
  • Terminal Bench 2.0：命令行操作能力
  • SWE Verified：软件工程任务解决能力
  • SWE Multilingual：多语言代码处理能力
  1. Search Agent（搜索智能体）
  • BrowseComp：网页浏览和内容理解
  • BrowseCompZh：中文网页浏览测试
  • HLE：可能指搜索相关的理解测试
  1. ToolUse（工具使用）
  • τ²-Bench：工具使用能力测试
  • MCP-Universe：多工具协作测试
  • MCP-Mark：工具使用标记测试
  • Tool-Decathlon：工具使用十项全能测试
    这些Benchmark共同构成了对大模型能力的全面评估体系，从语言理解、编程、数学到实际应用能力都有覆盖，帮助开发者了解模型的优劣势。
