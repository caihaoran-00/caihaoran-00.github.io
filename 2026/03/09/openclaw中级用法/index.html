skill检查者

安全要放在第一位，所以这个Skill Vetter（技能检查者）个人认为很必要，一样让openclaw自己去安装。（但使用中发现，如果你之前让Find Skills去找skills，然后没安装，下面再让openclaw安装，就不触发skill检查者了）

再推荐几个

  1. Summarize，用于总结 URL 或文件（网页、PDF、图片、音频、YouTube）。
  2. Find Skills，当你说”怎么做到 X”、”有没有做 X 的技能”时，它会帮你找并安装合适的技能。想扩展功能时用这个。（实测不太好用，没人肉去选的好）
  3. Humanizer，把 AI 写的文字改得更像人写的。编辑或审阅文本时用，去掉那些 AI 味儿的表达，比如过于华丽的词、套路化的句式、生硬的连接词等。
  4. Self-Improving-agent，自我反思 + 自我批评 + 自我学习 + 自我组织记忆。代理评估自己的工作，发现错误并永久改进。

例3：真●多Agent+群内交流+更完善的发文章到公众号

首先看一下多Agent模式与主 Agent 编排模式的区别：

维度 多Agent模式 主 Agent 编排模式
飞书机器人数量 多个（每个 agent 一个） 一个
群里的体验 多个机器人，各自响应 一个机器人，统一回复
配置复杂度 高（多个应用、多个凭证） 低（一个应用）
Agent 间协作 靠群聊对话（人类协调） 主 Agent 自动协调
适合场景 每个 agent 独立服务不同人/协调通信/长期专业化分工 主 agent 统筹多专家/临时任务
独立人格 ✅ 每个有自己的 SOUL.md ❌ 靠 prompt 模拟
持续优化 ✅ 有独立记忆文件 ❌ 每次重置
技能配置 ✅ 独立 TOOLS.md ❌ 继承主 Agent

这也就是为什么在这个例子中使用多Agent模式的原因。

我将创建5个Agent，也创建五个飞书机器人每个Agent拥有独立的workspace、人格设定和技能配置，能够持续累计领域经验，5个Agent将由主协调员调动。

cursor VS opencode VS openclaw

很多人还在把这些 AI 工具当成“更聪明的聊天框”，其实它们早就进化成能直接上手干活的“智能体”了。不管外壳长什么样，它们干活的逻辑其实就两招：改文件 和 敲命令行。

这三个工具到底有啥区别？

简单来说，它们就是你让 AI 接管电脑时的不同“操作界面”：

工具 定位 交互方式 核心体验
Cursor AI 代码编辑器 点点点 + 快捷键 写代码最丝滑，它直接长在你的 IDE 里。
OpenCode 终端命令行 AI 直接敲命令 极客的最爱，在终端里直接指挥它跑脚本、改配置。
OpenClaw 本地自动化助手 聊天软件（微信/TG） 像个后台管家，随时随地在手机上给电脑发指令。

它们是怎么干活的？

究其本质，它们的背后逻辑是一致的：
“先看看代码里写了啥（读文件/跑命令）” ➡️ “AI 脑子想一下” ➡️ “直接上手改代码或敲一行命令（写文件/执行）” ➡️ “看看结果对不对（观察输出）”。

即ReAct(Reason + Act):

Reason（思考）：也就是你说的“AI 脑子想一下”。模型不仅在聊天，它是在根据当前的状态（比如读到的代码报错、文件内容）规划下一步该干啥。

Act（行动）：这就是“直接上手改代码或敲命令行”。模型输出了一个具体的动作指令（比如 write_file 或 run_shell）。

Observe（观察）：这就是“看看结果对不对”。动作执行后的终端报错、文件更新后的语法检查结果，会重新喂给 AI，成为下一轮“思考”的上下文。

这是智能体（Agent）和普通聊天机器人的本质区别：机器人动嘴，它们会动手。
