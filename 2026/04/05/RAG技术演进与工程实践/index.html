RAG技术演进与工程实践

发表于 2026-04-05 更新于 2026-04-29 分类于 code 阅读次数： 本文字数： 8.6k 阅读时长 ≈ 31 分钟

写在前面

这篇文章想回答两个问题。

第一个问题：RAG 从最早的“向量库 + 大模型”一路发展到现在，中间到底出现了哪些技术？这些技术分别解决什么问题？

第二个问题：今天如果要做一个企业级 RAG 系统，文档解析、检索、数据库查询、权限、引用、评测这些模块应该怎么搭？哪些地方可以用框架，哪些地方最好自己控制？

本文会参考 Fareed Khan 的文章《Testing 18 RAG Techniques to Find the Best》。这类横向测试的价值不在于给出一个“世界第一”的排行榜，而是把常见 RAG 技术放在同一套实验里对比，适合拿来学习技术脉络。

先说结论：不要把里面的分数当成行业标准。Adaptive RAG 在那组实验里得分高，不代表所有场景都该上 Adaptive RAG；Semantic Chunking 在那组实验里得分低，也不代表它永远没用。RAG 的效果非常吃数据、问题类型、切分方式、评测集和模型。

一句话理解 RAG

RAG 的核心很简单：

它解决的不是“让模型变聪明”这个问题，而是“让模型回答时有资料可查”。这点很重要。很多 RAG 失败，不是模型不会写，而是系统根本没有把正确资料找回来。

一个最小 RAG 系统大概长这样：

这就是 Simple RAG。能跑，但很快会遇到问题：

  • 专有名词搜不到，比如设备型号、合同编号、政策编号
  • 文档切块把上下文切断
  • 表格、图片、扫描件解析不准
  • 检索结果相关但不够精确
  • 模型没有根据证据回答
  • 企业数据不只在文档里，还在数据库、工单、CRM 和业务 API 里
  • 权限、引用、审计、数据更新都没处理

后面所有 RAG 技术，基本都是围绕这些问题打补丁。

第一部分：RAG 技术经历了哪些阶段

阶段 0：没有 RAG，只有 Prompt 和微调

最早大家会把资料直接塞进 Prompt，或者用私有数据微调模型。

这两种方式都有问题。Prompt 放不下太多内容，微调又不适合频繁变化的知识。公司制度、产品文档、订单状态每天都可能变，拿这些东西去微调模型，维护成本太高。

RAG 出现后，思路变了：知识不一定要写进模型参数里，可以放在外部系统里，问的时候再取。

阶段 1：Simple RAG，先把资料找回来

Simple RAG 是最基础的形态：

它适合做最早的 baseline。无论后面要上多少高级技术，第一步都应该先跑一个 Simple RAG，看 bad case 到底在哪里。

Simple RAG 的优点是简单，问题也明显：只靠向量相似度容易漏掉精确词，chunk 切不好时会丢上下文，检索结果也没有二次判断。

阶段 2：先把“切块”和“上下文”修好

很多 RAG 问题不是检索算法的问题，而是文档在入库时就被弄坏了。

常见技术有：

技术 解决什么问题 怎么理解
Fixed-size chunking 快速建立 baseline 按长度切，简单但容易切断语义
Overlap chunking 防止上下文断裂 相邻 chunk 保留一段重叠文本
Semantic Chunking 按语义边界切分 听起来合理，但不一定稳定
Contextual Chunk Headers 补文档路径 给 chunk 加“文档名/章节/标题”
Sentence Window 命中一句话后扩展前后句 找得准，也保留上下文
RSE 补齐相邻片段 防止只取到中间一小段导致断章取义

这里有个实用判断：如果文档有清晰标题结构，优先保留标题路径；如果是合同、制度、手册，段落和条款编号往往比“语义分块”更可靠。

阶段 3：从“向量检索”走向“混合检索”

只用向量检索会有一个硬伤：它擅长语义相似，不擅长精确匹配。

比如用户问：

这里的 B18-甲型涡轮机 是精确词。向量检索可能觉得一堆“涡轮机维护规范”都很像，但真正需要的是包含这个型号的那段。

所以生产系统里常见的是混合检索：

相关技术：

技术 作用
BM25 抓关键词、编号、专有名词
Vector Search 抓语义相似
Hybrid Search 同时用关键词和向量
Fusion RAG / RRF 把多路召回结果合并排序

这也是为什么 Elasticsearch/OpenSearch 这类关键词索引在 RAG 系统里没有过时。向量库不是全部。

阶段 4：检索后再做“精排”

初召回通常会取 Top 20、Top 50，里面有不少噪音。Re-Ranker 的作用就是二次筛选。

常见做法是用 Cross-Encoder 类模型，让模型同时看 query 和候选片段，判断二者到底相关不相关。它比单向量相似度更准，但更慢，所以一般只用于候选集精排，不会拿来全库检索。

ColBERT 这类 Late Interaction 方法介于向量检索和 Cross-Encoder 之间，表达力更强，但索引和推理成本也更高。

阶段 5：改写问题，而不是只改资料

很多时候用户的问题太短、太口语，直接拿去检索效果不好。

相关技术有：

技术 怎么做 风险
Query Transform 把用户问题改写成更清晰的检索问题 改写错会带偏
Multi-query 生成多个检索 query 成本更高，结果要合并
HyDE 先生成一个假想答案，再用假想答案检索 假想答案如果偏了，会越找越歪
Doc2Query 给文档预生成可能被问到的问题 入库成本增加

我对这类技术的看法比较保守：可以用，但要评测。尤其是 HyDE，理论上很漂亮，但不是所有语料都受益。

阶段 6：让系统判断“检索结果够不够”

普通 RAG 是一次性流程：

但真实场景里，检索结果可能是错的、少的、互相冲突的。于是出现了 CRAG、Self-RAG、Feedback Loop 这类技术。

技术 核心思想
CRAG 先评估检索结果质量，不够好就纠正或兜底
Self-RAG 模型边生成边判断是否需要检索、证据是否足够
Feedback Loop RAG 根据反馈不断调整检索和生成
Adaptive RAG 先判断问题类型，再选择不同检索策略

这些技术的共同点是：RAG 不再是一条固定流水线，而是有了“判断”和“分流”。

阶段 7：从单文档问答走向结构化知识

当问题变成“跨文档总结”“多实体关系推理”“公司 A 和公司 B 的关系是什么”时，普通 chunk 检索就不够了。

于是出现了：

技术 适合场景
RAPTOR / Hierarchical RAG 长文档、多层摘要、宏观总结
GraphRAG 实体关系、跨文档推理、社区摘要
Knowledge Graph + RAG 强关系型知识、金融穿透、组织关系

RAPTOR 的思路是把 chunk 聚类，再层层摘要，形成树。用户问宏观问题时，不必只在叶子节点里找，可以检索上层摘要。

GraphRAG 则更偏关系网络。它把实体、关系、事件抽出来，再围绕图结构检索和生成。它的成本更高，不适合所有知识库。

阶段 8：多模态 RAG

企业文档不是只有文字。财报里有表格，论文里有公式，PPT 里有图，扫描件里有图片，图纸里有空间结构。

多模态 RAG 有两条路：

路线 做法 适合场景
转成文字 VLM/OCR 生成描述、表格、Markdown 大部分文档问答
保留视觉表征 直接对页面图片或 patch 建索引 图表、图纸、复杂版面

ColPali、VisRAG 这类方法就是第二类。它们不急着把页面转成纯文本，而是保留视觉布局去检索。代价是存储和检索成本更高。

第二部分：现在业界 RAG 系统怎么做

一个成熟系统大概长什么样

今天的企业 RAG 系统通常不是一个组件，而是一组模块：

这就是我理解的成熟 RAG。它不是一条链，而是一个能分流、能审计、能回放的问题处理系统。

推荐架构：先分流，再检索

这里的关键是查询路由。它本质上是意图识别：先判断用户问的是文档问题、指标问题、权限问题，还是多跳推理问题。

实现不一定靠大模型。常见做法是：

  • 规则：金额、订单、库存、客户、权限这类词先路由到结构化查询
  • Embedding 分类：用一批样例问题做类别原型，按相似度分类
  • 小模型/SLM：用 2B、7B 或 BERT 类模型做路由判断
  • LLM 复判：低置信度时让更强模型判断

我的建议是初期别训练太复杂的分类器。先用“规则 + SLM/LLM 结构化输出”跑起来，等业务稳定、样本足够，再把高频路由沉淀成轻量分类器。

质量闸门到底是什么

质量闸门不是一个固定产品模块，而是一组工程策略：在把答案交给用户前，判断当前证据够不够、风险高不高、要不要继续查或降级。

它不一定每次都很重。成熟系统通常会把检查分成轻重两层：

类型 检查什么 是否默认启用
轻量检查 检索相似度、引用数量、文档版本、SQL/API 是否成功 通常默认启用
重量检查 LLM-as-a-Judge、NLI 校验、人工审批 只在低置信度或高风险时触发

常见触发条件：

  • Top-K 相似度太低，或者没有找到能支撑答案的证据
  • 检索结果互相冲突，比如新旧制度同时命中
  • SQL/API 查询失败、超时或返回空结果
  • 问题涉及法律、财务、合规、客户赔付、医疗建议等高风险场景
  • 模型答案里有结论，但引用片段支撑不了这个结论

质量闸门后的动作一般有三种。

第一种是二次检索。比如第一次向量检索没找到，就改用 BM25 查专有名词，或者改写 query 后再查一次。二次检索不能无限循环，通常限制 1-2 次。

第二种是降级。系统承认证据不足，不硬编答案，比如返回“未找到足够证据，以下是相关条款供参考”，或者“当前数据库查询超时，请稍后重试”。

第三种是高风险审批。它不是所有请求都人工审核，只拦截少数高风险、低置信度、对外承诺类输出。普通内部问答不应该因为审批被卡住。

所以质量闸门会增加一点延迟，但可以通过分层控制：轻量检查默认跑，重检查按风险触发，所有检查都要有超时策略。它的目的不是让流程变复杂，而是防止系统在证据不足时自信地胡答。

文档解析层怎么选

LlamaIndex、LangChain 这类框架不等于文档解析工具。它们负责 RAG 编排，复杂文档解析最好交给专门工具。

常见选择：

工具 适合什么 注意点
LlamaParse LlamaIndex 生态、快速原型、复杂英文文档 商业 API / LlamaCloud 体系，成本要算清
Docling 本地化、批量 PDF/Office、可控 Pipeline 复杂表格要评估 VLM pipeline
MinerU 复杂 PDF、扫描件、公式、表格、中文文档 不同 backend 效果差异大
Unstructured 多格式文件拆成 elements 表格和复杂版面要调策略
Marker PDF 转 Markdown/JSON、本地批处理 企业使用前要看许可证
PaddleOCR / PP-StructureV3 中文 OCR、版面、表格、公式 更偏底层组件，需要集成
Azure Document Intelligence 云服务、表单、票据、企业 SLA 云厂商绑定和费用
Mistral OCR 云端 OCR、Markdown、表格、图片 bbox 适合快速接 API
Apache Tika 大量格式的文本和元数据抽取 不适合复杂版面还原

一个实用路由：

不要一开始就假设某个工具通吃。文档解析层最好做成可替换的 parser registry。

表格不要当普通文本处理

表格是 RAG 里最容易被低估的数据类型。普通文本只要判断“这段话和问题是否相关”，表格还要保住行列关系、表头、合并单元格、跨页结构和单元格坐标。

PDF 解析工具经常会把表格输出成 HTML、Markdown 或 JSON。这里要注意：HTML 更像结构保存格式，不一定是最终检索格式。

比较稳的做法是同时保存三份表示：

表示 用途
table_text / row_text 给 BM25 和向量检索用
table_html 保留结构、展示、高亮和溯源
table_json / csv / DataFrame 精确筛选、聚合和计算

小表可以整体作为 chunk。比如差旅报销标准表只有几行，就把表名、表头、行数据和来源页码一起入库。

中等表格可以按行组切分。每个行组都要带完整表头，否则“600 元/晚”这种值脱离列名就没意义。

大表和财务类表格不要整张塞给 LLM。几百行销售报表、库存表、回款表，应该转成 CSV、DataFrame 或 SQL 表。检索只负责找到相关表，具体筛选和计算交给 SQL 或 Python。

所以表格处理可以按这个原则走：

一句话：不要让 LLM 读 HTML 心算。HTML 用来保结构和溯源，检索用表格转写后的文本，计算用 DataFrame 或 SQL。

索引和存储怎么做

成熟系统通常是异构存储：

存储 放什么
向量库 语义检索
Elasticsearch / OpenSearch 关键词、过滤、元数据
关系库 / 数仓 / OLAP 订单、库存、指标、权限状态
对象存储 原始文件、图片、截图
图数据库 实体关系、GraphRAG

向量库不是数据库的替代品。涉及金额、库存、订单状态、绩效指标，一般先查数据库或业务 API，再让 LLM 组织语言。

增量更新、版本和 hash 是干什么的

文档不是一次入库就完事。制度会改，合同会补充，产品文档会更新，知识库页面也会反复编辑。RAG 系统要知道：哪些内容变了，哪些索引该更新，线上答案用的是哪个版本。

这里有三个概念：

概念 作用
版本 标记同一份文档的不同版本，比如 差旅制度_2026-04-01
hash 给文档、页面、段落或 chunk 算内容指纹，用来判断有没有变化
数据血缘 记录 chunk 从哪个文件、哪个版本、哪一页、哪个解析器来

小系统可以简单粗暴一点：上传新文档后删除旧 chunk，再全量重建。这个办法没错，适合文档少、更新少、成本可接受的场景。

但文档量一大，全删重建就不划算了。比如 200 页员工手册只改了一句话，如果全量重建，就要重新解析、切块、embedding、写向量库和关键词索引。真正变的可能只有 3 个 chunk。

更稳的做法是：

hash 的价值不是“不能全删”，而是减少无意义重建，并且保留回滚能力。企业系统通常还要保留历史版本，因为用户可能问“2025 年版本的制度当时怎么规定”。

数据血缘则是为了排查问题。一个 chunk 最好能追到这些信息：

这样线上答案出问题时，才知道它引用的是旧文档、新文档，还是某次解析出了错。

权限和溯源怎么做

企业 RAG 里，权限不能放到生成后再处理。检索前就要过滤。

每个 chunk 至少要带：

数据库查询也一样，要用只读账号、白名单表、字段权限、SQL AST 校验、超时限制。

权限过滤可以分两层。简单条件交给向量库的 metadata filter，比如租户、文档状态、版本、语言、文档类型、普通标签：

复杂权限不要全压给向量库。比如客户归属、组织层级、项目成员关系、字段级权限、复杂密级规则，最好先由权限服务或业务系统算出用户能看的 doc_id 范围，再让向量库只在这些文档里做语义检索。

一句话：向量库负责“在能看的内容里找最相关”，权限系统负责“判断哪些内容能看”。

系统最终返回的结果不应该只有一段自然语言。更稳的做法是返回“答案 + 结构化证据元数据”：

  • 引用文档
  • 页码或段落
  • bbox 高亮位置
  • SQL/API 查询条件与统计口径
  • 数据更新时间

这些字段应该来自检索系统、文档解析系统和数据库查询工具，不能让 LLM 自己编。LLM 负责把证据组织成自然语言；引用位置、bbox、SQL 条件、数据更新时间必须由系统真实返回。

没有这些，系统出错时很难排查。比如答案里的金额来自哪个表、按什么时间范围统计、是否只算已审批单据、数据是什么时候更新的，都需要能追溯。

怎么评测 RAG 系统

面试或方案评审时，可以这么说：

RAG 不能只评最终答案，要拆成三层。

第一层评检索。看 Top-K 里有没有正确证据。指标可以用 Recall@K、MRR、NDCG、Hit Rate。如果证据都没找回来，生成再强也没用。

第二层评生成。看模型是否基于证据答对，有没有编。指标可以看 Faithfulness、Answer Correctness、Citation Accuracy。关键样本最好人工抽检，LLM-as-a-Judge 可以辅助，但不要全信。

第三层评线上工程表现。看延迟、成本、SQL/API 成功率、权限拦截率、数据新鲜度、失败率和用户反馈。

最好维护一套 golden set：

注意，golden set 不应该绑定某一次 chunk_id。chunk 策略变了，chunk_id 会变；但标准证据对应的原文页码、段落、bbox、文档版本应该稳定。改 chunk 后，是重新评估新 chunk 是否覆盖标准证据，不是重建整个评测集。

LlamaIndex、LangChain、RAGFlow、Dify 怎么选

这几个不是同一类东西。

选择 定位 适合场景
LlamaIndex RAG 开发框架 文档问答、索引、检索、数据连接器
LangChain / LangGraph Agent 和工作流框架 工具调用、多步骤编排、状态机
RAGFlow 开箱 RAG 平台 企业知识库、文档问答、业务人员调试
Dify LLM 应用平台 workflow、chatflow、AI 应用搭建

如果是自己开发企业 RAG，我会优先用 LlamaIndex 做 RAG 主链路。它的抽象更贴近文档、节点、索引、检索器和查询引擎。

如果要做复杂 Agent、多工具调用、状态机，我会用 LangGraph。

如果业务目标是快速搭知识库产品，不想写太多代码，RAGFlow 更合适。

如果目标是做一堆 AI 应用和流程编排，Dify 更合适。

一句话：LlamaIndex 和 LangChain 是开发框架，RAGFlow 和 Dify 更像产品平台。它们不是完全替代关系。

不要一上来就做 Adaptive RAG、GraphRAG、Agentic RAG。先把最朴素的系统跑准。

第一步：做 baseline。

第二步：补结构化查询。

第三步：加评测和回放。

第四部分：18 种 RAG 技术速查

下面这张表不是排行榜，而是帮助理解技术分工。

技术 一句话解释 什么时候考虑
Simple RAG 最基础的向量检索问答 第一版 baseline
Semantic Chunking 按语义边界切块 文档结构弱时可以试
Contextual Chunk Headers 给 chunk 加标题路径 制度、手册、合同
Doc2Query 给文档预生成问题 用户 query 和文档表达差异大

HyDE 先生成假想答案再检索 query 太抽象时试验

RSE 补相邻片段 防止断章取义

Contextual Compression 压缩检索结果 上下文太长、噪音太多
Fusion RAG 多路召回融合 BM25 + 向量混合
Feedback Loop RAG 根据反馈修正答案 有用户反馈闭环时
Adaptive RAG 根据问题类型选择策略 query 类型差异大
CRAG 判断检索质量，不够好就纠正 检索结果不稳定
Self-RAG 生成时自我判断是否需要检索 高风险回答，但成本高
RAPTOR / Hierarchical RAG 聚类和层级摘要 长文总结、宏观问题
GraphRAG 实体关系图 + 社区摘要 跨文档关系推理
Multimodal RAG 文本、图片、表格一起检索 图表、扫描件、PPT

Fareed Khan 的测试可以作为学习入口，但生产选型还是要回到自己的数据。RAG 没有通用最优解，只有“你的问题卡在哪一层”。

结语

RAG 的演进其实很清楚。

一开始是“能不能找回来”。后来变成“能不能找准、排好、保留上下文”。再后来，问题变成“能不能处理表格、图片、数据库、权限、引用和数据更新”。到了企业场景，RAG 就不再是一个算法题，而是一个数据工程系统。

如果只记一件事，就记这个：

向量库只是其中一块。模型也只是其中一块。真正决定系统质量的，往往是那些不那么炫的东西：文档解析、权限、数据新鲜度、评测集、bad case 回放。

附录 A：新文件替代旧文件怎么设计

文档更新是企业 RAG 里很常见的事。比如公司重新上传了一版差旅制度，普通用户之后应该查到新版，而不是新旧版本混在一起。

一个容易出错的做法是：新文件一上传，就立刻删除旧 chunk，然后开始解析新文件。这个办法在小系统里能用，但有两个问题：

  • 新版本还没解析完时，用户可能查不到任何内容
  • 新版本解析失败时，旧版本已经删了，没法回滚

  • 历史追溯，比如“2025 年执行的差旅标准是什么”
  • 审计回放，比如排查某次历史回答为什么引用了旧制度

附录 B：RAG 系统关注哪些指标

RAG 的评测不能只看“答案好不好”。成熟系统会把问题拆开看：检索有没有找对、生成有没有基于证据、数据是不是最新、权限有没有越界、系统是否跑得动。

检索指标

检索指标依赖 golden set。每条样本要有问题、标准证据 doc_id/chunk_id/页码/原文片段。跑评测时，把系统返回的 Top-K 和标准证据对比。

指标 含义 怎么收集和计算 看它是为了什么
Recall@K Top-K 里是否覆盖标准证据 对每个问题取 Top-K，计算命中的标准证据数 / 标准证据总数，再求平均 判断系统有没有把资料找回来
Precision@K Top-K 里有多少是真相关 对每个问题人工或用标注集判断 Top-K 相关性，相关结果数 / K 判断检索结果噪音大不大
MRR 第一个正确证据排在第几 找到第一个相关结果的排名 r，记 1/r，所有问题求平均 判断正确资料是不是排得靠前
NDCG 排序质量 给结果标相关等级，按排名折损计算 DCG，再除以理想排序 IDCG 判断强相关证据是否排在弱相关证据前面
Hit Rate 是否命中标准证据 Top-K 里只要有一个标准证据就记 1，否则记 0，最后求平均 简单看“找没找到”

如果检索阶段没找到正确证据，后面的 LLM 基本只能靠猜。

生成指标

生成指标需要保存模型输出、引用证据、标准答案。可以人工标注，也可以用 LLM-as-a-Judge 做初筛，但高风险样本最好人工抽检。

指标 含义 怎么收集和计算 看它是为了什么
Faithfulness 答案是否被证据支持 把答案拆成若干结论，逐条判断是否能被引用证据支持，支持结论数 / 总结论数 防止模型编
Answer Correctness 答案是否正确 和标准答案比对，可人工打分，也可用 LLM judge 给 0-1 或 1-5 分 看最终回答是否解决问题
Citation Accuracy 引用是否真的支撑对应结论 对每个引用检查它是否支持答案中的那句话，正确引用数 / 总引用数 防止引用乱挂
Context Relevance 给模型的上下文是否相关 判断传入 LLM 的片段是否与问题相关，相关片段数 / 总片段数 判断检索和压缩是否带入噪音
Hallucination Rate 无依据结论比例 无证据支持的结论数 / 总结论数 衡量幻觉风险

生成指标可以用人工评，也可以用 LLM-as-a-Judge 辅助。但高风险样本最好人工抽检。

数据和版本指标

数据和版本指标来自索引流水线、文档元数据、数据库快照时间和答案 trace。

指标 含义 怎么收集和计算 例子
index_lag 源文档更新到索引生效的时间差 indexed_at - source_updated_at 09:00 上传，09:12 可检索，lag 12 分钟
version_match_rate 普通查询是否命中 active_version 普通查询中引用 active_version 的次数 / 总引用次数 100 次制度查询里 99 次命中新版
stale_answer_rate 答案引用旧版本证据的比例 引用 deprecated/historical 版本的答案数 / 总答案数 新制度已生效，但仍回答旧标准
data_lag 数据快照时间和回答时间的差距 answer_generated_at - query_snapshot_time 11:00 回答，用的是 10:45 快照
freshness_sla 新数据必须多久内可用 业务先定义 SLA，再统计超过 SLA 的比例 订单 5 秒内，制度 30 分钟内

这里的重点不是追求所有数据都实时，而是要知道答案基于哪个版本、哪个快照。如果只能查到 T+1 数仓，就应该在答案里说明“数据截至某时间”。

工程指标

工程指标来自线上日志和 trace。每次请求最好记录 request_id、用户、路由结果、检索耗时、SQL/API 耗时、LLM 耗时、token 用量、是否 fallback、是否权限拦截。

指标 含义 怎么收集和计算 看它是为了什么
端到端延迟 从提问到返回的总耗时 response_finished_at - request_started_at，看 P50/P95/P99 判断用户是否等得起
分阶段延迟 检索、重排、SQL、LLM 各自耗时 每个模块打 trace span，分别统计 P50/P95 定位慢在哪一层
SQL/API 成功率 结构化查询是否成功 成功调用次数 / 总调用次数，失败原因单独分类 发现权限、schema、超时问题
权限拦截率 被 RBAC/ABAC 拦截的请求比例 权限拦截次数 / 总请求数，也要看误拦截反馈 判断权限是否过宽或过严
单次请求成本 embedding、rerank、VLM、LLM、API 成本 统计 token、模型单价、API 次数、VLM 页数，折算到每次请求 防止复杂链路成本失控
fallback_rate 进入二次检索或降级的比例 fallback 次数 / 总请求数，并按原因拆分 判断主链路是否稳定

线上排障时，最好能回放完整请求链路：用户问题、路由结果、检索片段、SQL/API 调用、权限过滤、模型输入、模型输出、引用证据。没有这条链路，RAG 出错时很难判断是哪一层的问题。
