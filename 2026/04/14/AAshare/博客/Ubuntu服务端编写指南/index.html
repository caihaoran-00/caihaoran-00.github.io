Ubuntu服务端编写指南

这份文档只保留会直接影响服务端编码的关键信息，目标是指导实现一套适配当前 Duix-Mobile Android 客户端的 ASR + LLM + TTS 服务。

1. 总体方案

推荐采用“三服务、三环境、一个对外入口”的结构：

  • 对外只暴露 orchestrator
  • 内部拆分为 orchestrator、asr-service、tts-service
  • 每个服务各自一个 uv 虚拟环境
  • 后续 RAG 只接入 orchestrator

这样做的目的很简单：

  • Android 侧只需要配置一个地址
  • ASR/TTS/LLM/RAG 的职责边界清晰
  • 各服务依赖互不污染
  • 后续替换模型或扩容时改动最小

5. 服务职责

5.1 orchestrator

职责：

  • 接收 Android 请求
  • 如果有音频则调用 asr-service
  • 调用远端 LLM API
  • 按句切分 LLM 输出
  • 调用 tts-service
  • 用 SSE 把 text/audio/done/error 回传给 Android

这是 Android 唯一需要知道的服务。

5.2 asr-service

职责：

  • 接收整段语音
  • 返回识别文本

第一版建议直接做“整段识别”，和当前 Android 的录音上传方式最匹配。

5.3 tts-service

职责：

  • 接收一整句文本
  • 返回可直接喂给 DUIX 的 PCM 音频

建议一条请求只合成一整句，不要一次喂很长段落。

7. 音频格式约定

服务端必须和当前 Android 客户端保持一致：

  • 16000 Hz
  • 16bit
  • mono
  • raw PCM

也就是说：

  • asr-service 最好直接接收这套格式
  • tts-service 最终也必须输出这套格式

如果 TTS 原生输出不是这套格式，应当在 tts-service 内部完成重采样和格式转换，不要把这一步留给 Android。

8. 推荐模型

结合你当前目标，推荐：

  • ASR：SenseVoiceSmall
  • TTS：Fun-CosyVoice3-0.5B

如果第一版更想先求简单，也可以：

  • ASR：paraformer-zh 或 SenseVoiceSmall
  • TTS：MeloTTS 作为轻量备选

9. uv 环境使用方式

  • orchestrator/.venv
  • asr-service/.venv

10. 编码顺序

推荐按下面顺序写：

  1. asr-service
    先完成“PCM -> 文本”
  2. tts-service
    再完成“文本 -> 16k/16bit/mono PCM”
  3. orchestrator
    最后把 ASR -> LLM -> TTS -> SSE 串起来

这样调试最省事，因为问题会更容易定位。

11. 启动顺序

推荐启动顺序：

  1. 启动 asr-service
  2. 启动 tts-service
  3. 启动 orchestrator
  4. 再让 Android 指向 orchestrator

12. 实现时需要注意的点

12.1 模型常驻

  • ASR/TTS 必须在服务启动时预加载
  • 不要每次请求再加载模型

12.2 先用单 worker

如果 ASR/TTS 共用同一块 GPU，第一版先都用 1 worker，不要急着开多个 uvicorn workers，否则很容易重复占显存。

12.3 LLM 按句切分再 TTS

推荐流程：

  • LLM 流式输出
  • orchestrator 按句切分
  • 每句单独调用一次 tts-service
  • 每句 TTS 一出来就立刻通过 SSE 回给 Android

这是最适合数字人播报的方式。

12.4 Nginx 不要缓冲 SSE

如果后面加 Nginx，记得关闭 SSE 缓冲，否则客户端会感觉一直不返回。

13. 后续接 RAG 的位置

后面如果接 RAG，只加在 orchestrator 即可：

不要把 RAG 逻辑塞进 asr-service 或 tts-service。

14. 第一版验收标准

第一版只要满足下面几点，就算跑通：

  • Android 只连 orchestrator
  • 语音输入能经 ASR -> LLM -> TTS 完整走通
  • orchestrator 能稳定返回 text/audio/done
  • Android 端能驱动数字人开口说话

做到这一步之后，再考虑并发优化、RAG、监控和自动拉起。
