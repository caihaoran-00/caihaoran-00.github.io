    发表于 2026-05-07 更新于 2026-05-11 分类于 code 阅读次数： 本文字数： 6.1k 阅读时长 ≈ 22 分钟

    前言

    本文整理 HB.AI 私域分型与督导系统 V1 的技术方案。核心目标是把企业微信群里的用户反馈转化为可结构化、可追踪、可运营的分型、风险、复购/升级信号和后台待办。

    正文

    企业微信群内用户状态识别、分型判断与运营辅助技术方案

    一、项目背景

    加速燃正在从“快速减重产品”转向“AI 驱动的激素能量支持品牌”。后续业务重点不只是卖单一产品，而是围绕用户的长期状态、节律、能量、饮食秩序、复购升级和私域运营，建立一套可以持续运转的系统。

    客户现阶段已经有大量企业微信群。群内每天会产生用户反馈、打卡、效果质疑、购买咨询、升级咨询和风险反馈。过去这些内容主要靠运营人员人工查看、判断和跟进。随着群数量增加，人工会被大量“中间劳动”拖住：谁是下午崩盘型，谁是晚间失控型，谁有复购机会，谁出现负面情绪，谁需要人工介入，这些判断如果长期依赖人工，很难规模化。

    因此，HB.AI 第一阶段不适合做成一个泛化聊天机器人，也不适合一开始建设完整经营中枢。更合理的路径是先落到企业微信群这个高频场景里，跑通一个最小可用闭环：用户在群里 @AI，系统理解用户反馈，更新用户状态，给出简短督导回复，并把风险、复购、升级和待跟进事项同步到运营后台。

    第一阶段的目标不是替代所有运营人员，而是先替代大量重复、分散、依赖经验判断的中间工作，让用户状态可见，让运营知道下一步该跟谁、为什么跟、怎么跟。

    V1 只处理企业微信群内用户 @AI 的消息，不默认读取所有群消息，也不做 AI 私聊。

    群内建议保留人工运营或督导官。AI 负责中间判断和初步回复，人工负责控场、异常兜底、复购升级承接和用户信任维护。

    2.3 技术需求

    V1 采用可控的 Workflow / Pipeline，而不是自由发挥式 Agent。

    2.4 数据与运营需求

    V1 必须有结构化状态库和基础运营后台。

    三、先说结论

    第一阶段不建议直接采用“自由发挥式 Agent”作为主系统，也不建议把 HB.AI 做成一个泛化聊天机器人。

    更稳的实现方式是：

    自研 Workflow / Pipeline 控制主流程，LLM 作为分类、抽取、判断和回复生成能力使用。

    这里的 Workflow / Pipeline 指后端固定执行的一套处理流程，避免 AI 自己决定下一步做什么。LLM 负责理解用户消息和生成结构化结果，后端负责校验、写库、生成待办和决定后续动作。

    5.3 Workflow Engine 与 quick_analyze 的关系

    quick_analyze 不是独立流程控制器，而是 HB Workflow 在同步链路中调用的一次 LLM 分析服务。

    更准确地说：

    Workflow 决定每一步必须做什么，quick_analyze 只负责在其中完成快速语义分析，包括消息分类、关键信息抽取、初步风险判断、复购/升级信号初判和回复草稿生成。

    7.1 用户 @AI 后的同步链路

    V1 不要把所有分析都串行执行，否则延迟会很大。

    同步链路只做用户当下回复必须依赖的事：

    目标是让用户在 2-5 秒内 收到回复。

    7.2 同步与异步边界

    同步链路只处理用户当场回复必须依赖的步骤；异步链路处理更深的分析和运营沉淀。

    任务 执行方式 原因
    识别用户与群 同步 回复前必须知道是谁、在哪个群
    读取用户历史状态 同步 回复需要上下文
    quick_analyze 同步 需要生成当场回复
    群内回复 同步 用户体验要求
    保存原始消息与回复 同步 保证基础留痕
    创建高健康风险人工待办 同步 避免高风险漏接
    更新分型证据分 异步 不影响当场回复
    更新主分型 异步 需要综合历史证据
    案例候选沉淀 异步 不影响用户体验
    用户摘要更新 异步 可以延迟
    数据看板刷新 异步 不需要实时

    八、异步链路

    用户收到回复后，后台继续做更深的分析。

    但异步链路不是对每条消息都完整执行所有分析，而是通过 Async Job Router 按消息类型和信号触发对应任务。这样可以避免无效调用，也能控制成本和延迟。

    8.1 异步任务路由

    条件 执行任务
    有状态反馈 更新分型证据分
    有普通健康风险、商业风险或负面反馈 生成 / 归并风险待办
    有复购信号 生成复购待办
    有升级信号 生成升级待办
    有典型分型或风险事件 判断案例候选
    达到阶段节点 做阶段复盘
    普通闲聊 只记录，不跑深分析

    8.2 为什么要异步

    因为这些任务不一定要立刻完成：

    异步后，前台回复快，后台分析深。

    九、用户状态与分型

    9.1 用户状态字段

    每个用户至少维护一份状态。

    V1 暂按 user_id + group_id 维护用户状态。同一用户出现在多个群时，先视为不同群内状态；后续如果需要跨群统一身份，再拆出 group_memberships。

    9.2 分型不是每句话都直接改

    分型采用“证据分”机制。

    不要让主分型频繁跳变。

    9.3 V1 分型类型

    分型 识别信号 策略
    下午崩盘型 下午困、三四点没精神、想吃甜 午后续航观察
    晚间失控型 晚上想吃、晚饭后破功 夜间收口观察
    嘴馋乱吃型 不饿也想吃、零食冲动 区分真饿和乱饿
    疲惫低能量型 没劲、疲惫、动力差 低摩擦能量支持
    高预期错配型 怎么还没瘦、没效果、想快点 降低承诺，人工关注
    升级潜力型 想系统一点、后面怎么继续 标记升级信号
